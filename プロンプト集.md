### ステアリングファイルを設定

#### Step3. ステアリングファイルを編集

```markdown
---
inclusion: always
---
# AI エージェント動作制御

本ファイルは、Kiro（AI エージェント）の「振る舞いの方針」を定める。

## 公式ドキュメント確認
- AWS の仕様（Organizations / SCP 等）は、公式ドキュメントを確認して根拠（出典）を示す。

## 推測禁止
- 不確かな点を推測で断定しない。根拠が示せない場合は「未確認」と明示する。

## 破壊的操作・実適用の禁止
- 本ワークショップはレビュー専用。実 AWS 環境への適用・デプロイ・破壊的操作は行わない。
- 成果物は「適用せずレビューのみ（ドライラン的アプローチ）」を前提に提示する。

## 不明点は確認
- 要件・前提が不明な場合は、勝手に補完せず、確認のための質問を返す。
```

#### Step4. ステアリングファイル設定後、チャット

```text
SCPを本番アカウントにアタッチして有効化するAWS CLIコマンドを実行して
```

### 既存の組織要件定義書を詳細化

#### Step1. チャットにて、組織要件定義書を詳細化

```text
あなたはAWSガバナンス設計を支援する専門家です。
2026_0615_Kiro_Workshop_Organization フォルダのsample-organization-requirements-generic.pdf を読んで「Vibeドラフト」を作ってください。
- 背景・全体方針（開発スピードと統制のバランス）を前提として簡潔にまとめる
- アクセス制御／ログ・監査／コスト・課金／セキュリティ の4区分で、対象と統制意図を整理する
- 環境軸・組織軸・基盤の役割・統制の段階も、OU設計の手がかりとして別枠で整理する
- 曖昧な点や抜けは「論点」として挙げる
EARS化やトレーサビリティ表はまだ作らないでください（次の Spec モードで行います）。軽い整理に留め、その出発点になるドラフトとしてまとめてください。
```

#### Step3. 新しい要件定義書を修正する

```text
論点の7は、タグポリシーで拒否するようにして
```

### Spec モードで仕様駆動開発

#### Step1. Spec モードで仕様駆動開発

```text
あなたはAWSガバナンス設計を支援する専門家です。#vibe-draft.md を基に、AWS Organizations のOU/SCP設計の Spec を作成してください。まず requirements フェーズを作ります。
※時間短縮のため、今回は「セキュリティ」区分のみを対象にします。
requirements に含めること:
1. セキュリティ区分の内容を構造化（各項目に 識別子（例: REQ-SEC-01）・区分・対象・統制意図）。代表的な要件を1〜2個に絞る
2. 各項目を EARS 形式の正式要件（THE … SHALL … / WHEN … / WHERE … / IF … THEN …）に整える
3. 実装方針の制約（例 REQ-IMPL-01）: 最終的に CloudFormation（IaC）で表現し、静的検証のみ・デプロイしない（レビュー専用）
この段階では OU/SCP の確定設計やIaCコードは作らず、EARS 要件を requirements.md にまとめてレビューできる状態にしてください。
```

## 📝 補足

- 各プロンプトはコードブロックの **右上のコピーアイコン（📋）** でそのままコピーできます。`<...>` で囲まれた箇所は、自分の成果物に置き換えてください。
- `#[[file:...]]` や `#ファイル名` は Kiro 固有の構文で、指定したファイルの内容をチャットの文脈に読み込みます。
- すべての成果物は **レビュー専用** です。実 AWS 環境への適用・デプロイは行いません。
